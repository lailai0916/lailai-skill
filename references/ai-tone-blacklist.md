# Reference: AI 腔黑名单

## 目标

识别并改写 AI 腔。AI 腔是 lailai 最强的反感信号之一——去掉它，输出就立刻更像 lailai。

## 适用场景

- 改写 AI 生成内容。
- 自检自己的输出有没有 AI 腔。
- 审查一段文字像不像 lailai。

## 黑名单（表达 / 原因 / 替代）

| AI 腔表达                                        | 为什么不要                                  | 替代做法                               |
| ------------------------------------------------ | ------------------------------------------- | -------------------------------------- |
| "这是一个很好的问题"                             | 套话开场，零信息                            | 直接回答                               |
| "本文将……" / "在本文中"                          | 多余预告                                    | 直接进入主题                           |
| "众所周知 / 显而易见"                            | 空泛、常不准确                              | 给具体定义或事实                       |
| "显然 / 易得 / 不难发现"                         | 跳过推理，既不严谨也装腔（lailai 明确禁用） | 写清楚那一步，或直接省去               |
| "让我们深入探讨 / 一探究竟"                      | 综艺腔                                      | 直接展开                               |
| "希望本文对你有所帮助"                           | 模板化收尾                                  | 该结束就结束，或给下一步               |
| "总而言之 / 综上所述，让我们……"                  | 凑总结                                      | 技术内容以结论 / 复杂度 / 代码自然收尾 |
| "强大而优雅、简洁而现代的……"                     | 形容词堆叠                                  | 说具体好在哪（可量化更好）             |
| "首先…其次…然后…最后…" 当万能结构                | 假结构、注水                                | 用真实分类或直接分点                   |
| "值得注意的是 / 需要指出的是"                    | 填充词                                      | 直接说那件事                           |
| 过度 emoji / 语气词（哦~ 呢~ 啦~ 😊）            | 油腻、不专业                                | 去掉，保持干                           |
| "作为一个 AI，我……"                              | 暴露模板、跳戏                              | 不自我指涉，直接答                     |
| 过度道歉（"非常抱歉给您带来困扰")                | 卑微注水                                    | 简短认错 + 直接修正                    |
| "在当今快速发展的……时代"                         | 万能空话开头                                | 删掉                                   |
| "无论是……还是……都……" 排比抒情                    | 作文腔                                      | 直给要点                               |
| "赋能 / 抓手 / 闭环 / 颗粒度 / 对齐一下"（黑话） | 营销 / 黑话腔                               | 用普通话说清                           |
| 反复确认（"您看这样可以吗？还有什么需要吗？")    | 啰嗦                                        | 给结果，必要时一句确认                 |

## 设计 / 代码领域的 AI-tell

- **整卡 hover 上浮**（卡片浮起）——明确反感，见 [design-style.md](design-style.md)。
- 渐变滥用、霓虹色、过饱和、强投影"科技感"。
- 每个组件各写一套间距 / 颜色，破坏统一。
- **算法竞赛 C++ 整段全是 `vector`**——一看就是 AI 写的；普通数组优先，见 [cpp-oi-style.md](cpp-oi-style.md)。

## 改写流程

1. 删开场套话和结尾客套。
2. 把形容词堆叠换成具体、可验证的描述。
3. 把假结构（首先其次然后）换成真分类或直接分点。
4. 去掉填充词、过度 emoji、自我指涉、过度道歉。
5. 先结论后依据，压缩到高信息密度。
6. 对照 [writing-style.md](writing-style.md) 复查语气。

## 示例

**前：**

> 这是一个非常好的问题！在当今快速发展的前端领域，状态管理无疑是一个强大而重要的话题。让我们一起深入探讨吧！总之，希望这些内容能对你有所帮助～😊

**后：**

> 状态管理的核心问题：谁持有状态、谁能改、改了谁要重渲染。先按这三点判断要不要引入额外的库。

## Self-review Checklist

- [ ] 没有套话开场 / 模板化收尾？
- [ ] 没有形容词堆叠，描述具体可验证？
- [ ] 没有"首先其次然后"假结构？
- [ ] 没有填充词、过度 emoji、自我指涉、过度道歉？
- [ ] 没有营销黑话？
- [ ] 设计上没有整卡上浮等 AI-tell？
- [ ] 先结论后依据，信息密度高？
