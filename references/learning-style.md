# Reference: 学习与知识整理风格

## 目标

模拟 lailai 的学习方式与知识整理习惯。部分 `待验证`，但与思考方式、写作风格、站点"参考资料"习惯一致，可信度中等。

## 适用场景

- 整理笔记、知识点、题解、文档。
- 规划一个新主题怎么学、怎么记。
- 审查一份学习材料是否符合 lailai 的整理习惯。

## 核心规则

1. **先定义再展开**：学一个概念先界定它是什么、不是什么，再看用途、推导、例子。
2. **重来源**：非平凡结论给参考资料（OI Wiki、Wikipedia 中 / 英、OEIS）。把"标注来源"当默认，不是可选。
3. **结构化、分层次**：知识整理成"定义 → 分类 → 推导 → 例题 → 复杂度 / 边界"的骨架，而非流水账。内容多时**按逻辑分层归组**（先归类、再往各组填要点），不平铺一长串同级小标题——"多而不乱"靠层次，不靠堆量。但反过来也别**过度切碎**：不是每个小点都配个标题，能一段话讲顺的就别拆成密集小节，求**通俗易懂**，结构服务于易读。
4. **可复用、可检查**：把稳定的规律写成规则 / 模板固化下来（站点的 `.claude/rules/` 即此习惯外化）。
5. **以题 / 例驱动**：抽象知识配具体例题；算法知识点末尾挂 `## 例题`。
6. **精确**：数量、范围、复杂度都精确化、标清楚。
7. **最小可用 + 迭代**：先把核心理解跑通，再逐步补全细节，而不是一次追求完美笔记。
8. **整合融合而非追加**：已有某主题笔记、再拿到一份新材料时，先通读两边，按"取精去糟"融成一份——重合处合并、互补处归位、新内容插进合理章节，而不是在原笔记末尾新增一块塞进去。与 skill 维护的"合并 > 新增"（见 [maintenance-guide.md](maintenance-guide.md)）是同一条原则。

## 推荐做法

- 新主题先搭骨架（要点分类），再逐格填充。
- 内容多先分层归组，再往每组填要点；同级标题超过五六个就考虑上一层归类。
- 拿到新材料先融进既有笔记的合理位置，不另起一段堆在末尾。
- 每个知识点问："定义清楚了吗？有没有例子？边界 / 复杂度是什么？"
- 把反复用到的模式抽成模板（代码模板、题解模板、笔记模板）。
- 记录来源链接，方便回溯与核对。

## 不推荐做法

- 抄概念不写自己的理解，不配例子。
- 笔记写成无结构的大段流水账，或一长串平铺、不分层的同级小标题。
- 反过来过度切碎：什么都加小标题、堆密集格式，把本可连贯的内容拆得不易读。
- 拿到新材料整块追加到旧笔记末尾，不做研读与融合。
- 不标来源、把猜测当事实记下来。
- 一次性追求"完美笔记"而迟迟不开始。

## 示例：一则知识点笔记骨架

```markdown
## 前缀和

**前缀和**（Prefix Sum）—— 定义、用途。

### 定义

$S_k=\sum_{i=1}^{k}a_i$。

### 用途

区间求和从 $O(n)$ 降到 $O(1)$。

### 例题

- ……

### 参考资料

- [前缀和 - OI Wiki](https://oi-wiki.org/basic/prefix-sum/)
```

## Self-review Checklist

- [ ] 概念有清晰定义（是什么 / 不是什么）？
- [ ] 结构张弛有度、通俗易懂——内容多时分层归组，但不流水账、不平铺一堆同级标题，也不什么都切成密集小标题？
- [ ] 新材料是研读后融进合理位置，而非末尾整块追加？
- [ ] 配了例子 / 例题？
- [ ] 标了来源（非平凡结论）？
- [ ] 数量 / 范围 / 复杂度精确？
- [ ] 反复出现的模式抽成了模板？
