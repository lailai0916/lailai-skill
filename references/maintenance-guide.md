# Reference: 维护与进化指南

## 目标

说明如何长期维护、验证、进化 lailai.skill，让它随时间越来越像 lailai，而不是越来越复杂。

## 适用场景

- 发现新习惯 / 偏好 / 反感项时。
- 定期复查 skill 是否仍准确。
- 跑一轮优化、记录版本、清理过时规则。

## 核心闭环

skill 的进化是一个固定闭环。每个新习惯都走完这条链,不跳步:

```text
observation → rule → example → test → evolution log →(回到)→ observation
```

1. **observation**:新习惯先进 [../observations.md](../observations.md)（带证据、标稳定性）。观察 ≠ 规则。
2. **rule**:多次重现且稳定后,沉淀进 `../profile/` 或本目录某 reference 的"核心规则"。
3. **example**:在 `../examples/` 加一个 before/after,展示新规则下"像 / 不像"的对照。
4. **test**:在 `../tests/` 加可判定的检查项,让这条规则能被验证。
5. **evolution log**:在 [../evolution/evolution-log.md](../evolution/evolution-log.md) 记一条(为什么改、改了什么、测试结果、保留 / 回滚),重要的同步 [../CHANGELOG.md](../CHANGELOG.md)。
6. 回到 observation:用真实任务继续观察,发现偏差再进入下一轮。

闭环的意义:任何规则都可追溯到一条观察,也都有对应示例和测试。**没有示例和测试的规则是半成品**。

## 进化机制（11 步）

1. **观察记录**：新发现先写进 [../observations.md](../observations.md)。观察 ≠ 规则。
2. **稳定性判断**：区分短期情绪、临时偏好、长期稳定习惯。一次性现象不直接成文。
3. **规则沉淀**：长期、稳定、可迁移、可执行的观察整理进 `profile/` 或 `references/`。
4. **示例更新**：规则变化影响输出时，更新 `examples/`。
5. **测试更新**：新规则可验证时，更新 `tests/`。
6. **达尔文式优化**：评估 → 改进 → 测试 → 保留或回滚（见 [../prompts/evolve-skill.md](../prompts/evolve-skill.md)）。
7. **版本记录**：重要变化记入 [../CHANGELOG.md](../CHANGELOG.md)。
8. **进化日志**：每次重要进化记入 [../evolution/evolution-log.md](../evolution/evolution-log.md)（原因、修改、测试、保留与否）。
9. **过时规则清理**：定期检查旧规则是否仍符合 lailai 当前状态，过时则标记 / 修改 / 删除。
10. **真实项目验证**：在个人网站、README、Docusaurus 文档、代码、设计审查、聊天模拟等真实任务中验证。
11. **人工最终确认**：长期人格、价值判断、隐私边界、公开表达的重要变化必须人工确认。

## 进化原则

- 观察 ≠ 规则；临时情绪 ≠ 长期人格；一次性偏好 ≠ 稳定习惯。
- 新习惯必须验证后才进正式规则。
- 规则必须可执行、可检查。
- **先解码意图，再落规范**：lailai 说"记进 lailai-skill""加这条"时，别把原话直接抄进去。先弄清他真正想表达什么（含糊就先确认），再按本 skill 的写作 / 用词 / 排版规范与规则格式，写成可执行、易懂的条目——原话是素材，成文要规范化。
- 每次进化都能说清"为什么改"，并尽量配套示例或测试。
- 若新规则让输出更不像 lailai，回滚。
- 若旧规则与当前状态冲突，更新或废弃。
- 复杂度必须服务准确性，不为显得高级而加结构。
- **以身作则**：skill 自身的每个文件都按它定义的写作 / 排版 / 结构规范来写；改 skill 时顺手让被改文件符合规范，发现违规即修。

## 反膨胀：收敛而非累积

目标是"越来越像 lailai"，不是"越来越大"。素材越喂越多，规则数量应**收敛**而非线性增长——风格是有限的，前若干篇就覆盖了大部分模式。铁律（每次维护都适用）：

1. **判重优先**：加一条规则前，先确认现有规则覆盖不了它。覆盖得了 → 不加，顶多用更好的证据 / 例子**替换**已有内容（长度不变）。
2. **净增趋零**：每次维护报告"净增行数"。素材越多，净增应越接近 0；持续为正是警报——在累积，不是收敛。
3. **软上限**：单个 `references/*.md` 目标 ≤ 150 行，`profile/*.md` ≤ 80 行，每个 Self-review Checklist ≤ 8 项。到顶后只能压缩 / 替换，不能继续追加。
4. **合并 > 新增**：两条规则讲同一件事 → 合并成一条。宁可一条更准，不要两条各说一半。
5. **observations.md 是缓冲不是仓库**：已沉淀的条目标记后定期删除，不长期堆积。

任何让文件变长却没让输出更像 lailai 的改动，回滚。

## 观察 → 规则的判定

一条观察可以沉淀为规则，当且仅当全部满足：

- **稳定**：在 ≥ 2～3 个不同场景重现。
- **可迁移**：不只适用一次性情境（若只适用某项目，标明适用范围，归为局部规则）。
- **可执行**：能写成"做 X、不做 Y"，可被检查。
- **不涉隐私**：不含账号 / 密钥 / 他人敏感信息。
- **非情绪**：不是某次心情的产物。

不满足的，留在 `observations.md` 继续观察。

## 局部规则 vs 长期规则

- **长期 / 可迁移规则**：写入 `profile/` 或通用 `references/`（如 writing-style、ai-tone-blacklist）。
- **局部 / 项目规则**：标明适用范围（如 docusaurus-style 只适用站点；数学分隔符站内用 `$...$`）。冲突时局部规则在其范围内优先。

## 定期复查清单（建议每次较大更新或定期跑一次）

- [ ] `待验证` 项是否拿到真实样例可以确认 / 修正？
- [ ] 有没有规则已与 lailai 当前状态冲突？标记 / 改 / 删。
- [ ] `examples/` 与 `tests/` 是否跟上了最新规则？
- [ ] `observations.md` 里有没有已稳定、可沉淀的条目？
- [ ] 用 [../evolution/evaluation-rubric.md](../evolution/evaluation-rubric.md) 抽查几个真实输出打分。
- [ ] 跑一轮合并剪枝：找重复 / 过时 / 可合并的规则并压缩；确认本轮**净增行数 ≤ 0 或接近 0**，文件未超软上限。
- [ ] 重要变化记了 `CHANGELOG.md` 与 `evolution-log.md` 吗？

## 需要 lailai 补充的资料（长期）

- 真实日常聊天样例（校准 conversation-style、daily-chat-style）。
- 真实决策 / 取舍样例（校准 decision-rules）。
- 明确的反例（"这个不像我"），比正例更有信息量。
- 新的写作 / 代码 / 设计作品（持续更新风格基线）。

## Self-review Checklist

- [ ] 新发现先进了 `observations.md`，而不是直接改规则？
- [ ] 沉淀的规则满足稳定 / 可迁移 / 可执行 / 不涉隐私 / 非情绪？
- [ ] 局部规则标明了适用范围？
- [ ] 改动配了示例 / 测试，能说清"为什么改"？
- [ ] 变差的改动回滚了？
- [ ] 重要变化记了日志，敏感变化做了人工确认？
